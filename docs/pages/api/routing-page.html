Introduction

Routing in a web application is about managing state. Modern single-page web applications make use of the HTML5 history API in order to retain normal browser navigation while not forcing a complete refresh of the page on every change. Footwork incorporates history.js to help smooth over the browser history API inconsistencies.

The router included with footwork has a few key features:

  • Unlimited nesting of routers.
  • Unlimited named dynamic view outlets per router.
  • Context based sub-routing.
  • Filtered/predicate based routes.

In essence the included router in footwork combines several design patterns (MVC + MVVM + Observable) and context-based subrouting to provide a novel and extremely powerful routing solution.

Usage

Routers have two principle components which make them useful:

  • Routing configuration/definition used to define routes, controllers, and their actions.
  • <outlet name=""></outlet> declarations which are the output points your router controls.

Configuration

Routers are incorporated into footwork viewModels and are a part of their configuration definition.

Outlets

viewModels 'wrap' a section of your DOM, providing the binding points and logic for that section of your HTML. In a similar sense a router provides all routing logic for that section of the DOM. If anywhere within in the DOM there is an <outlet name=""></outlet> declaration, footwork will travel up the element node path searching for the nearest parent viewModel which has a router. That router is able to control what view/component is displayed at any one time in any <outlet name=""></outlet> declarations found within its scope.
